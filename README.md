## 4. 遇到的问题及解决方法
**问题 1：VS Code 终端提示“无法将 git 项识别为 cmdlet...”**
* **原因**: 系统没有安装 Git 环境，或者环境变量未配置生效。
* **解决方法**: 前往官网下载并安装 Git，随后彻底重启 VS Code，使其加载最新的系统环境变量。

**问题 2：提交时提示 "Author identity unknown"**
* **原因**: 没有配置全局的用户邮箱和用户名，Git 出于协作规范拒绝提交。
* **解决方法**: 运行 `git config --global user.email "我的邮箱"` 和 `user.name "我的名字"` 补充身份信息。

**问题 3：推送代码时遭遇 "Failed to connect to github.com port 443" 超时错误**
* **原因**: 国内网络环境连接 GitHub 存在波动或被屏蔽。
* **解决方法**: 尝试连接手机热点；或灵活变通，改用 GitHub 网页端的 "upload file" 和在线新建文件功能完成最终的作业提交。

## 6. 读书笔记（选做）：从“写代码”到“做工程”的思维跃迁

**阅读书籍**: 《构建之法：现代软件工程》（邹欣 著） / 《代码大全》（Code Complete）

**阅读感悟**: 

在本次读书实践周中，结合 Git 版本管理工具的学习，我系统阅读了软件工程领域的经典著作。如果说大一、大二的课程教会了我们如何编写正确的逻辑，那么这本关于“软件工程”的书，则重塑了我对“如何构建一个健壮系统”的认知。

**1. 告别“小工作坊”：版本控制与代码资产化**
书中提到“软件=程序+软件工程”，这句看似简单的话击中了我之前的开发痛点。在过去学习和独立复现一些核心算法（例如手动实现 KNN 算法或是 SM4 加密逻辑）时，我往往处于一种“单打独斗”的思维模式。代码只要能跑通测试用例就算成功，文件版本往往用“v1”、“v2_最终版”这样粗放的方式命名。

然而，随着近期参与到更复杂的大型项目中（如担任数据集核心模块开发等团队角色），我深刻体会到这种“工作坊式”开发在团队协作中的脆弱性。结合本次 Git 实践，我认识到每一次标准的 Commit 并不是简单的文件备份，而是项目演进的“时间机器”。清晰的提交说明、合理的分支（Branching）策略，能够将个人的代码片段转化为团队可以共享、回溯和审查的“数字资产”。Git 迫使我在写完代码后多思考一步：这段代码修改了什么？为什么修改？这正是工程师与纯编码者的区别之一。

**2. 接口即契约：全栈视角的模块化思维**
在探讨系统架构时，书中对“高内聚、低耦合”的强调让我对前期的全栈开发经历有了新的反思。在实际的开发中，无论是使用 FastAPI 搭建后端接口，还是进行前端 UI/UX 的高保真面板设计，前后端的对接往往是最容易产生摩擦的环节。

读书让我意识到，API 接口不仅是数据传输的通道，更是不同模块负责人之间的“契约”。一个好的设计，应该像 Git 的合并机制一样，既允许各分支（各模块）独立、自由地演进，又能在最终合并时保证主干的稳定。如果在设计之初没有做好数据流向和异常处理的约定，后期的联调就会变成灾难。这要求我们在写下第一行代码前，必须具备全局的架构视野。

**3. 持续重构与自我迭代**
书中的另一个核心观点是软件的生命周期并非终结于发布，而是始于维护。无论是探索机器学习与 NLP 领域的语义图谱，还是进行底层网络协议的分析，最初写出的代码往往都是为了验证思路的“原型”。优秀的代码是经过不断重构和优化打磨出来的。Git 提供的版本回退和差异对比功能，正是我们敢于重构烂代码的最大底气，因为它消除了“改坏了找不回来”的恐惧。

**总结**
本次读书实践周是一次理论与实践的完美印证。书籍为我提供了“为什么要规范化工程”的理论高度，而 Git 的实操则给了我“如何落地规范”的底层工具。在未来的专业学习与项目实战中，我将不再仅仅满足于算法的跑通，而是要严格遵循工程化标准，写出不仅机器能读懂，而且团队队友也能读懂的优雅代码。。
